3.1 JavaScript - Beginner to Senior

Core mental model

JavaScript is a garbage-collected, prototype-based, dynamically typed language standardized by ECMAScript. The language itself defines execution semantics; browsers and Node.js provide host APIs and event-loop integration. 

Source Code → Parse → AST → Bytecode / JIT Optimization
                      │
                      ▼
              Execution Contexts
                      │
                 Call Stack
                      │
        Host APIs / libuv / browser runtime
                      │
           Task queues + Microtasks
                      └────► Event Loop ────► Call Stack

Must-master syllabus

Primitive/reference semantics; coercion; equality; scope; lexical environment; TDZ; hoisting; closures; this; call/apply/bind; prototypes; classes; property descriptors; iterables/iterators; generators; promises; async/await; event loop; modules; errors; memory/GC; functional programming; OOP; immutability; currying; composition; debounce/throttle; deep clone limitations.

BeginnerWhat is an execution context and what is stored inside it?
An execution context is the runtime environment in which JavaScript code executes. It contains the variable environment, lexical environment, scope chain, and this binding. The global context is created first; each function call creates a function execution context. Contexts are pushed onto the call stack. During creation, declarations are registered; during execution, statements run.
IntermediateExplain hoisting precisely. Are let and const not hoisted?
All declarations are processed before normal execution. var is initialized to undefined; function declarations are initialized with their function value. let, const, and class are also hoisted but remain uninitialized in the Temporal Dead Zone until declaration evaluation. Access during the TDZ throws ReferenceError.
IntermediateWhat is a closure, and where have you used one in production?
A closure is a function together with access to the lexical environment in which it was created, even after the outer function returns. Production uses include factory functions, memoization, private state, event handlers, custom React hooks, debouncing, middleware configuration, and callbacks. Mention the memory-retention trade-off: captured objects remain reachable while the closure remains reachable.
AdvancedExplain the event loop including microtasks and macrotasks.
Synchronous JavaScript runs on the call stack. Async operations are handled by the host runtime. When the stack is empty, the runtime drains the microtask queue-Promise reactions and queueMicrotask-before processing the next task such as a timer or I/O callback. In browsers, rendering opportunities occur between tasks. A recursively growing microtask chain can starve timers and rendering.
AdvancedHow does prototypal inheritance work internally?
Objects have an internal [[Prototype]] link. Property lookup checks the object first and then follows the prototype chain until a property is found or the chain ends at null. Constructor functions expose a prototype object used as the prototype of instances created with new. ES classes are syntax over prototype-based behavior, with additional semantics such as strict mode and private fields.
IntermediatePromise.all vs allSettled vs race vs any?
Promise.all fulfills when all fulfill and rejects on the first rejection. allSettled waits for every input and returns statuses. race settles with the first settled input. any fulfills with the first fulfillment and rejects with AggregateError only if all reject. Choose based on failure semantics, not convenience.
SeniorWhat can cause memory leaks in a JavaScript application?
Unremoved event listeners, forgotten timers, unbounded caches, detached DOM trees, accidental globals, long-lived closures retaining large objects, unresolved subscriptions, and queues that grow without backpressure. Diagnose with heap snapshots, allocation timelines, retained-size analysis, and production memory metrics.
CodingImplement debounce and explain the closure. 
function debounce(fn, delay) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

The returned function closes over timer. Each call resets the timer, so only the latest invocation executes after inactivity. Time O(1) per call; auxiliary space O(1), excluding captured arguments.

3.2 TypeScript - Beginner to Senior

Core mental model

TypeScript is a statically typed superset of JavaScript. It performs static analysis at build time and then erases types to emit plain JavaScript, so type safety does not exist at runtime unless you add explicit runtime validation. 

TypeScript Source → Parse → Type Check → Control Flow Analysis → Emit JavaScript
                                         │
                                         ▼
                               tsc / Babel / SWC pipeline
                                         │
                                         ▼
                           JavaScript runtime (browser or Node.js)

Must-master syllabus

Type inference; annotations; structural typing; interfaces vs types; unions; intersections; literal types; narrowing; discriminated unions; generics and constraints; utility types; mapped types; conditional types; indexed access; keyof; template literal types; declaration files; module resolution; tsconfig strictness flags; unknown vs any; never; variance basics; and runtime validation boundaries.

BeginnerWhat problem does TypeScript solve, and what does it not solve?
TypeScript catches many category errors at compile time: wrong argument shapes, unsafe property access, and invalid unions. It does not enforce those types at runtime because emitted JavaScript has no type metadata by default. External data from APIs, forms, and files still needs runtime validation.
Intermediateany vs unknown vs never?
any disables type safety and allows any operation. unknown is a safer top type: you must narrow or assert before using it. Prefer unknown for untrusted values, then apply type guards or schema validation. never is the bottom type: it represents code that cannot be reached or a function that never returns. It is useful for exhaustive checks and signaling impossible states.
IntermediateInterface vs type alias: how do you choose?
Both can model object shapes. Interfaces support declaration merging and are often preferred for public object contracts and class implementations. Type aliases are more flexible for unions, intersections, tuples, and conditional types. In practice, choose for readability and consistency, not ideology.
AdvancedExplain discriminated unions and exhaustive checking.
A discriminated union uses a shared literal field such as kind to represent variants. Control-flow narrowing picks the correct branch by checking that field. Exhaustive switch handling plus never checks helps ensure newly added variants are not silently ignored.
AdvancedHow do generic constraints improve API design?
Generics preserve relationships between inputs and outputs, while constraints like T extends { id: string } guarantee required capabilities. This keeps APIs reusable without losing correctness, and reduces unsafe casts in calling code.
SeniorHow would you migrate a large JS codebase to TypeScript safely?
Migrate incrementally: enable allowJs, start with strict checks in leaf modules, define shared domain types, and gate unsafe boundaries with runtime validation. Track any-usage, CI type errors, and defect trends. Avoid a single big-bang rewrite that blocks product delivery.
SeniorWhy can "type-safe" code still fail in production?
TypeScript types describe assumptions at compile time, but runtime data can violate them. Network payload drift, feature flags, malformed JSON, and environment differences can all break behavior. Senior practice combines static typing with runtime schema checks, observability, and defensive error handling.
CodingImplement a generic pluck utility and explain type safety. 
function pluck<T, K extends keyof T>(items: T[], key: K): T[K][] {
  return items.map((item) => item[key]);
}

K extends keyof T restricts keys to valid property names of T. The return type T[K][] preserves the exact value type for that key, preventing invalid property access at compile time.

Must master: component model, JSX, props/state, controlled inputs, lifting state, reconciliation, keys, Fiber, render/commit phases, hooks rules, useState batching, effects and cleanup, stale closures, refs, context, reducers, custom hooks, memoization, error boundaries, lazy/Suspense, transitions, profiling, virtualization, accessibility, forms, testing, SSR/hydration, Server/Client Components.

IntermediateVirtual DOM vs reconciliation vs Fiber—what is the difference?
The virtual DOM is the in-memory representation of UI elements. Reconciliation is React's process for deciding what changed between renders. Fiber is the internal architecture and work-unit data structure that enables interruptible rendering, priorities, scheduling, and concurrent features. They are related but not synonyms.
IntermediateWhat causes a React component to render?
Its state changes, its parent renders, a consumed context value changes, or an external store subscription signals an update. A render does not necessarily mean DOM mutation; React reconciles the output and commits only required changes. React.memo can skip parent-driven renders when props are shallowly equal, but local state and context can still trigger rendering.
IntermediateuseMemo vs useCallback vs React.memo?
useMemo caches a computed value, useCallback caches a function reference, and React.memo can skip rendering a component when props compare equal. Memoization has overhead and should target measured expensive work or reference-stability requirements, not be applied everywhere.
AdvancedWhy are keys important? Why is array index often dangerous?
Keys identify sibling elements across renders. Stable keys let React preserve component identity and state. Index keys become incorrect when items are inserted, deleted, sorted, or filtered because state may attach to the wrong logical item. Index is acceptable for static lists that never reorder or change identity.
SeniorRedux Toolkit vs Context vs Zustand—how do you choose?
Context is best for relatively low-frequency cross-tree values such as theme or authenticated user metadata. Redux Toolkit is appropriate for complex shared client state requiring predictable transitions, middleware, DevTools, normalization, and large-team conventions. Zustand offers a smaller API and selective subscriptions for simpler global client state. Server-state fetching should usually use RTK Query, TanStack Query, or framework caching rather than hand-built global state.
AdvancedExplain RTK Query cache invalidation.
RTK Query cache invalidation is an automated mechanism that forces query endpoints to re-fetch stale data when server-side changes occur. It relies on a declaration-based "cache tag" system to track which queries rely on which pieces of data, ensuring the application UI remains synchronized with the database.
SeniorHow would you diagnose a slow React page?

Here is a systematic approach to diagnosing a slow React page, useful for both real debugging and interview discussion.

  1. Confirm what "slow" means first. Before touching code, identify the exact symptom:
    • Slow initial load (TTFB, bundle size, parsing).
    • Slow interaction (button click feels laggy).
    • Janky scrolling/animation (dropped frames).
    • Slow re-renders (typing in an input lags).
    Each symptom has a different diagnosis path, so do not guess.
  2. Measure before you optimize.
    • Chrome DevTools Performance: Record while reproducing slowness, inspect flame graph blocks, and look for long tasks (>50ms) blocking the main thread.
    • React DevTools Profiler: Record a render, inspect which components re-rendered and why, and flag high render counts relative to visible UI change.
    • Lighthouse / Web Vitals: Use LCP, INP, and CLS to identify initial-load and responsiveness issues.
  3. Check common culprits first.
    Symptom Likely cause
    Whole app re-renders on small state change State too high in tree, missing memoization, or context value recreated on each render.
    Typing in input is laggy Expensive computation on each keystroke or uncontrolled re-renders of large lists.
    Initial load is slow Large bundle, render-blocking resources, or too many synchronous API calls before first paint.
    List scrolling is janky Rendering thousands of DOM nodes without virtualization.
    Gets slow after some usage Memory leak from uncleared intervals, subscriptions, or event listeners.
  4. Apply fixes matched to the cause.
    • Unnecessary re-renders: Use React.memo, useMemo/useCallback where profiling proves value, move state closer to usage, and split contexts.
    • Expensive computation: Memoize, or move work off the main thread with useDeferredValue, startTransition, or Web Workers for heavy workloads.
    • Bundle size: Code-split with React.lazy and verify large dependencies with a bundle analyzer.
    • Large lists: Use virtualization (react-window, react-virtual).
    • Network waterfall: Parallelize independent fetches, move data fetching earlier, and cache server state (for example with React Query).
CodingBuild a production-grade useFetch hook. What concerns matter?
Handle aborting stale requests, race conditions, loading/error/data state, dependency stability, retries only where safe, authentication expiry, deduplication, cache policy, and unmount cleanup. In production, prefer a mature server-state library unless the use case is intentionally small.

Common React traps

  • Using useEffect for derived state that can be computed during render.
  • Missing effect cleanup for subscriptions/timers.
  • Suppressing dependency warnings instead of fixing stale closures.
  • Mutating state.
  • Using index keys for dynamic lists.
  • Globalizing server state unnecessarily.
  • Memoizing everything without measurement.
Rapid revision: Be able to explain exactly why a component rendered, when effects clean up, how stale closures occur, how RTK Query cache keys/tags work, and when Context is not a Redux replacement.

5. Next.js

IntermediateSSR vs SSG vs ISR vs CSR?
SSR renders per request; SSG pre-renders at build time; ISR serves cached static output and regenerates according to revalidation semantics; CSR renders/fetches primarily in the browser. Choose per route and data requirement. Consider freshness, personalization, cacheability, SEO, latency, build time, and infrastructure cost.
AdvancedServer Components vs Client Components?
Server Components execute on the server, can access server-side resources directly, and do not ship their component JavaScript to the browser. Client Components are required for state, effects, event handlers, and browser APIs. A 'use client' boundary pulls the client module graph into the client bundle; keep it narrow.
SeniorExplain Next.js caching without hand-waving.
Next.js caching is a multi-layered system. The CDN caches static assets and ISR pages according to Cache-Control headers and revalidation settings. The server runtime may cache SSR output in memory or on disk, depending on deployment. Data fetching hooks like getServerSideProps or RTK Query can implement their own caching strategies. Cache invalidation is explicit via revalidation intervals, tags, or manual purges.
AdvancedHydration mismatch causes?
Mismatches occur when the server-rendered HTML does not match the client-rendered output. Causes include non-deterministic rendering (random values, dates, Math.random), browser-only APIs in server code, conditional rendering based on client state, and differences in environment (e.g., locale, time zone). Use useEffect or useLayoutEffect for client-only logic and ensure deterministic output for SSR.

Production checklist

Performance: Use next/image for optimized images, next/script for third-party scripts, and analyze bundle size with next build --profile. Prefer ISR or SSG for cacheable pages. Use React Profiler to identify slow components.

Version trap: Next.js caching and rendering semantics evolve. State which version and router you actually used before answering implementation-specific questions.

6. Node.js & Express

IntermediateIf Node.js is single-threaded, how does it handle concurrency?
JavaScript callbacks execute on the main event-loop thread, but the runtime delegates many operations to the OS or libuv. Network I/O is generally event-driven; some filesystem, DNS, crypto, and compression work uses libuv's thread pool. Worker Threads provide JavaScript parallelism for CPU-bound work.
AdvancedExplain Node.js event-loop phases.
The event loop has multiple phases: timers (setTimeout/setInterval), pending callbacks (I/O callbacks deferred to the next loop), idle, poll (I/O polling and executing callbacks), check (setImmediate), and close callbacks. Microtasks (Promise reactions, queueMicrotask) run after each phase before moving to the next. Understanding these phases helps reason about callback ordering, timers, and I/O behavior.
AdvancedStreams and backpressure?
Streams are abstractions for reading/writing data in chunks. They can be readable, writable, or duplex. Backpressure occurs when the consumer cannot keep up with the producer. Node.js streams signal backpressure by returning false from write() and pausing the readable stream until the consumer drains. Properly handling backpressure prevents memory bloat and ensures smooth data flow.
SeniorCluster vs Worker Threads vs Child Processes?
Cluster: Multiple Node.js processes sharing the same port, each with its own event loop and memory. Good for scaling I/O-bound workloads across CPU cores. example: cluster.fork(). Worker Threads: Multiple threads within a single process, sharing memory. Suitable for CPU-bound tasks that need shared state. example: new Worker(). Child Processes: Separate OS processes that can run any executable, communicate via IPC(Inter-Process Communication). Useful for isolation or running non-Node.js code. examples: cluster.fork(), new Worker(), and child_process.fork(). 
const { Worker } = require('worker_threads');
const worker = new Worker('./worker.js');
worker.on('message', (msg) => {
     console.log('Message from worker:', msg);
});
FAANGHow would you scale an API to millions of requests?
  • Use a load balancer to distribute traffic across multiple instances.
  • Implement horizontal scaling with stateless services.
  • Use caching (CDN, Redis) to reduce database load.
  • Optimize database queries and use read replicas.
  • Implement rate limiting and throttling to protect against abuse.
  • Monitor performance metrics and set up auto-scaling based on load.

7. Backend, APIs, Authentication & Security

IntermediateJWT vs session-based authentication?
Session-based authentication stores a session ID on the server and a cookie on the client. JWT-based authentication encodes claims in a signed token that the client sends with each request. Sessions can be invalidated server-side; JWTs are stateless but require careful expiry and revocation strategies. Both can be secure if implemented correctly, but JWTs are often used for stateless APIs and microservices.
BeginnerAuthentication vs authorization?
Authentication is the process of verifying a user's identity, while authorization determines what actions or resources the authenticated user is allowed to access. In other words, authentication answers "Who are you?" and authorization answers "What can you do?"
AdvancedHow do XSS, CSRF, CORS, and SQL injection differ?
XSS (Cross-Site Scripting) is a client-side attack where malicious scripts are injected into web pages viewed by other users. CSRF (Cross-Site Request Forgery) tricks an authenticated user into performing unwanted actions on a web application. CORS (Cross-Origin Resource Sharing) is a browser security feature that restricts cross-origin HTTP requests. SQL injection is a server-side attack where malicious SQL code is inserted into queries, potentially compromising the database.
SeniorDesign retries correctly.
Retries should be applied only to idempotent operations. Use exponential backoff with jitter to avoid thundering herds. Set a maximum retry count and consider circuit breakers for persistent failures. Log and monitor retry behavior to detect systemic issues.
AdvancedWhat is idempotency?
An operation is idempotent if performing it multiple times has the same effect as performing it once. In HTTP, GET, PUT, and DELETE are idempotent methods, while POST is not. Idempotency is important for safely retrying requests without unintended side effects.
IntermediateREST API design checklist?
Use nouns for resources, pluralize collections, use HTTP methods correctly (GET, POST, PUT, DELETE), support filtering/sorting/paging via query parameters, version your API, return appropriate status codes, and provide consistent error responses. Consider HATEOAS for discoverability and document your API with OpenAPI/Swagger.

Security revision checklist

Must know: XSS, CSRF, CORS, SQL injection, parameterized queries, input validation, output encoding, authentication vs authorization, JWT vs session, HTTPS/TLS, password hashing (bcrypt, Argon2), rate limiting, logging sensitive data, and secure headers (CSP, HSTS).

Rapid revision: Understand the OWASP Top 10, secure coding practices, and how to defend against common web vulnerabilities.

8. PostgreSQL, MySQL & MongoDB

IntermediateWhat is a database index internally?
A database index is a data structure (commonly a B-tree or hash) that allows the database to quickly locate rows based on indexed columns. It stores key-value pairs where the key is the indexed column value and the value is a pointer to the corresponding row in the table. Indexes improve read performance but add overhead for writes and storage.
AdvancedWhy can an index make performance worse?
Indexes can slow down write operations (INSERT, UPDATE, DELETE) due to the overhead of maintaining the index structure. They also consume additional disk space and memory. If a query does not filter on indexed columns or if the index is not selective, the database may choose a full table scan instead, making the index unused and adding unnecessary overhead.
SeniorPostgreSQL vs MySQL vs MongoDB for your fintech platform?
PostgreSQL offers strong ACID compliance, advanced SQL features, and robust indexing, making it suitable for complex financial transactions. MySQL is widely used, has good performance, and is easier to set up, but may lack some advanced features. MongoDB is a NoSQL document database that provides flexibility in schema design and horizontal scaling, but may require additional effort to ensure data integrity and consistency. The choice depends on the specific requirements of the fintech platform, such as transaction volume, data complexity, and scalability needs.
AdvancedExplain ACID and isolation anomalies.
ACID stands for Atomicity, Consistency, Isolation, and Durability. Atomicity ensures that a transaction is all-or-nothing. Consistency ensures that a transaction brings the database from one valid state to another. Isolation ensures that concurrent transactions do not interfere with each other. Durability ensures that once a transaction is committed, it will remain so, even in the event of a crash. Isolation anomalies include dirty reads, non-repeatable reads, and phantom reads, which can occur when transactions are not properly isolated.
IntermediateWhat is the N+1 query problem?
The N+1 query problem occurs when an application executes one query to retrieve a list of entities (N) and then executes an additional query for each entity to retrieve related data (+1 for each). This results in N+1 total queries, which can lead to performance issues. Solutions include eager loading, batch fetching, or using joins to retrieve all necessary data in a single query.

  2. Inspect query plan.
  3. Check row estimates vs actuals.
  4. Reduce scanned rows and returned columns.
  5. Design index for predicates/order/join.

  7. Retest with production-like cardinality.
  8. Monitor after rollout.

FAANGDesign a credit report processing platform integrating Equifax and Experian.
Start with requirements: request volume, SLA, latency, provider limits, consistency, audit requirements, PII handling, retries, and whether results can be asynchronous. Use an API gateway/auth layer, orchestration service, durable job queue, provider adapters, encrypted data stores, webhook callback processor, idempotency store, dead-letter queue, observability, and admin UI. Never expose provider credentials to clients. Use correlation IDs and immutable audit events. Separate transient retryable errors from permanent business failures.
SeniorHow do you make a webhook receiver production-safe?
Authenticate signatures, preserve raw request body when required for HMAC verification, enforce replay windows, use idempotency keys/event IDs, respond quickly after durable acceptance, process asynchronously, retry safely, keep a dead-letter path, log correlation IDs without leaking secrets/PII, and provide reconciliation jobs for missed events.
IntermediateWhen should you use a queue?
When work can be asynchronous, needs smoothing during traffic spikes, requires durable retries, isolates dependencies, or fans out to consumers. Trade-offs: eventual consistency, duplicate delivery, ordering limitations, operational complexity, poison messages, and observability requirements.
AdvancedExplain CAP theorem without the common oversimplification.
In the presence of a network partition, a distributed data system must choose between continuing to provide every request with a response that may not be consistent (availability) or rejecting/delaying some requests to preserve a single consistent view (consistency). CAP is about partition scenarios, not a permanent two-of-three menu for all normal operation.

System design interview framework

  1. Clarify functional and non-functional requirements.
  2. Estimate users, QPS, storage, bandwidth, and growth only where useful.
  3. Define API contracts and data model.
  4. Draw the simplest viable high-level design.
  5. Trace critical read/write flows.
  6. Identify bottlenecks and failure modes.
  7. Deep dive into consistency, cache, queue, database, security, and observability.
  8. State trade-offs and evolution path.

Hard CodingHard coding drill: LRU Cache
Use a hash map from key to node plus a doubly linked list ordered by recency. get moves a node to most-recent position; put updates/inserts and evicts least-recent when capacity is exceeded. Both operations are O(1) average time; space O(capacity). Common mistakes: not updating existing keys, broken prev/next links, capacity 1, and returning stale values.
